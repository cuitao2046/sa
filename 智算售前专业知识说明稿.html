第一章　核心认知：训练与推理的本质差异

智算售前工作中最常见的误区，是在没有厘清客户场景之前就开口推荐硬件。训练（Training）和推理（Inference）对算力、显存、互联带宽的需求截然不同，是两条几乎平行的需求曲线。任何一套"通用最强方案"，在两类场景下往往都不是最优解。掌握这条基本逻辑，是所有选型和报价的第一步。

1.1 训练（Training）的核心特征

训练是指通过大量数据样本迭代更新模型权重的过程。从硬件视角看，训练的本质是一个计算密集 + 通信密集 + 长时间稳定运行的计算任务，对硬件的要求可以从以下六个维度理解：

  • 算力需求极高：大模型预训练通常要求 FP16/BF16/FP8 精度下的高算力（TFLOPS/PFLOPS级），百亿到千亿参数模型需要千卡~万卡 GPU 并行协作。
  • 显存需求庞大：训练过程中 GPU 显存需要同时存放模型权重（FP16/BF16）、梯度（与权重同等大小）、FP32 权重副本（master weights，混合精度训练用于精度更新的全精度备份）、优化器状态（AdamW 一阶/二阶动量各占4字节×参数量）以及前向激活值。一个 70B 参数模型用 AdamW 优化器的混合精度训练，理论总显存约 70B×16字节 = 1120 GB。
  • 卡间通信带宽是瓶颈：反向传播过程中需要将梯度在所有参与训练的 GPU 上进行 All-Reduce 同步，这要求机内 GPU 之间（NVLink/UB）和跨机 GPU 之间（IB/RoCE）都具备超高带宽和低延迟，否则 GPU 会因等待通信而大量闲置，导致"加速比"远低于 1。
  • 网络要求无损且大规模：梯度同步是突发式大流量通信，网络丢包会导致 RDMA 重传，严重降低集群效率。因此训练集群参数面网络要求零丢包，采用 InfiniBand 原生无损机制，或 RoCEv2 加 PFC/ECN/DCQCN 拥塞控制机制实现无损以太。
  • 长时间连续稳定运行：一次大模型预训练通常需要数周到数月，对集群可靠性和故障自愈能力提出极高要求。Meta 在训练 Llama3-405B 的 54 天内统计了 419 次故障，其中光模块故障和网络故障是主因。
  • 核心 KPI 是 MFU（Model FLOPs Utilization）：理想情况下 MFU = 100%，意味着 GPU 的理论算力被充分利用。实际上 30~50% 已属良好，原因在于通信、数据加载、内存访问都会造成算力空闲。

1.2 推理（Inference）的核心特征

推理是指将训练好的模型部署上线，对外提供实时预测/问答服务的过程。推理与训练的最大区别在于：推理不需要存储梯度和优化器状态，显存压力大幅降低；但对单位成本（每 Token 成本）、响应延迟（TTFT首 Token 时延）和高并发吞吐（TPS/QPS）更加敏感。

  • 显存带宽决定速度上限：生成式推理（自回归解码）每生成一个 Token，都需要把整个模型权重从显存读入一次，因此推理速度几乎直接受显存带宽约束。

每秒生成 Token 数（TPS）≈ 显存带宽（TB/s）÷ [参数量（B）× 精度字节（2/1/0.5）]

  • 显存容量决定能不能跑：如果模型权重放不进单卡显存，必须多卡并行，而多卡通信开销会抵消带宽优势。因此"能不能在给定卡上运行"是推理选型的第一关口。
  • KV Cache 是长上下文的核心成本：KV Cache 的大小 = 2 × 层数(L) × 头数(H) × 头维度(D) × batch大小(B) × 序列长度(seq) × 精度字节。支持 128K 长上下文时，KV Cache 可消耗数 GB 至数十 GB 的额外显存，这是 H200（141GB）和 B200（192GB）相比 H100（80GB）更受推理青睐的根本原因。
  • 批处理（Batching）是提升吞吐的关键：多个请求共享一次模型权重读取，可以把显存带宽收益分摊到更多 Token，提升整体吞吐。Continuous Batching 技术让在途请求动态进入批次，是当前主流推理框架（vLLM/TGI）的核心优化。
  • 能效比（TFLOPS/W）是推理侧的核心采购维度：同等算力下功耗更低的硬件，在大规模推理集群中能显著降低电费和制冷成本。国产昇腾 910B 约 400W 对比英伟达 H100 700W，能效比优势在推理场景尤为突出。

1.3 显存需求快速估算方法

这是售前现场选型的第一个计算动作。掌握以下估算逻辑，可以在 30 秒内给出"能跑/不能跑/需要几张卡"的初步判断。

1.3.1 推理显存估算

推理显存（GB）≈ 参数量（B）× 精度字节 × 1.2（安全系数）

精度字节：FP32 = 4B，FP16/BF16 = 2B，INT8 = 1B，INT4 = 0.5B。安全系数 1.2 覆盖框架开销。

1.3.2 训练显存估算（以 AdamW 优化器为例）

训练显存（GB）≈ 参数量（B）×（2+2+4+4+4）字节 + 激活值

分项说明：①模型权重 FP16/BF16 = 参数量×2字节；②梯度 FP16/BF16 = 参数量×2字节；③FP32 权重副本（master weights）= 参数量×4字节；④ AdamW 一阶动量 = 参数量×4字节；⑤ AdamW 二阶动量 = 参数量×4字节。合计约 16字节/参数（2+2+4+4+4=16，不含激活值）。70B 参数模型训练最小显存约 70×16 = 1120 GB，需要 H100×14张（或 H200×8张）。

💡 实战提示：客户现场说"我要训练 70B 的模型"，标准回答是：FP16 混合精度下最少需要 14 张 H100（1120GB），建议用 2节点×8卡 H100，再叠加 KV Cache、框架开销后保守起见上 16 卡起步。

第二章　英伟达 GPU 硬件详解

英伟达在全球 AI 芯片市场中占据约 70% 份额（IDC 2025.5），其 GPU 产品线是智算售前最核心的硬件知识。本章按照产品代际顺序，详细介绍 A100（基准参考）、H100（当前主力）、H200（大显存旗舰）、B200（2025 旗舰）、GB200（Grace Blackwell 超芯片）、GB300（Grace Blackwell Ultra 推理旗舰）六款主流训练/推理卡及超芯片的技术规格与选型要点，以及面向中国市场的合规版本。

A100 是 2020 年发布的 Ampere 架构产品，尽管已不是最新代，但其装机量仍是目前全球最大，是理解后续架构演进的基准参考点。

2.2 H100（Hopper 架构 / 4nm）——当前训练标杆

H100 是 2022 年发布、2023 年大规模交付的 Hopper 架构旗舰，是全球 AI 大模型训练的主力机型，也是所有售前方案中最常出现的核心硬件。相比 A100，H100 引入了若干重大突破：

  • 全新第四代 Tensor Core：首次原生支持 FP8 精度，FP8 算力约 2 PFLOPS（稠密），相比 A100 FP16 提升约 6.4×，在推理侧可大幅降低显存占用和计算成本。
  • HBM3 显存升级：80GB HBM3，带宽提升至 3.35 TB/s，相比 A100 HBM2e 的 2.0 TB/s 提升约 67%，推理 Token 速度相应提升。
  • NVLink v4 互联提升至 900 GB/s：机内 8 卡全互联带宽从 600GB/s 提升至 900GB/s，对张量并行（TP）的效率提升显著。
  • 引入 Transformer Engine：根据层的数据分布动态选择 FP8/FP16 混合精度，在保证精度的前提下最大化利用 FP8 算力优势，是 H100 在大模型训练上相比 A100 有代际领先的核心原因。

2.3 H200——大显存旗舰，专为大模型和长上下文设计

H200 是在 H100 芯片基础上升级了 HBM 内存子系统的强化版本，于 2023 年发布、2024 年批量交付。核心改变只有一处，但影响深远：HBM3e 容量从 80GB 翻至 141GB，带宽从 3.35TB/s 提升至 4.8TB/s。

这一升级对三类场景意义重大：①能装下更大的单卡模型（减少分片通信开销）；②KV Cache 容量翻近两倍，支持更长的上下文窗口（如 128K Token）；③推理吞吐（TPS）相比 H100 提升约 60%（显存带宽直接提升了 TPS 上限）。

2.4 B200（Blackwell 架构）——2025 旗舰

B200 是 NVIDIA 2024 年发布、2025 年逐步量产交付的全新 Blackwell 架构旗舰 GPU，采用双 die 封装，单卡集成两个 GPU die，在算力、显存、互联三条轴同步大幅升级。

  • 192GB HBM3e 显存（8.0 TB/s 带宽）：相比 H100 提升 2.4×，相比 H200 提升 1.7×，单卡可装下更大模型，推理 TPS 的上限大幅抬高。
  • FP4 精度支持：Blackwell 引入第五代 Tensor Core，首次支持 FP4 精度，FP4 推理算力约 9 PFLOPS（是 FP8 的 2×），在可接受精度损失的推理场景下计算效率极高。
  • NVLink v5（1.8 TB/s）：机内卡间互联带宽从 v4 的 900GB/s 翻倍至 1.8TB/s，对 MoE 等需要高频卡间通信的模型架构提升显著。
  • GB200 = Grace CPU + B200×2：NVIDIA 推出了 Grace-Blackwell 超芯片 GB200，把一颗 Grace ARM CPU 和两颗 B200 GPU 通过 NVLink-C2C（900GB/s）集成在一个封装上，实现 864GB 统一内存空间，是 NVL72 机柜方案的基础单元（详见 2.5 节）。

2.5 GB200（Grace Blackwell 超芯片）——机柜级超节点基础单元

GB200 是 NVIDIA 2024 年发布的 Grace Blackwell 超芯片（Superchip），将 1 颗 Grace ARM CPU（72 核 Neoverse V2）与 2 颗 B200 GPU 通过 NVLink-C2C（900 GB/s）互联封装在一起。GB200 不是独立 GPU 卡，而是超节点机柜的基础构建模块——36 颗 GB200 组成一个 NVL72 机柜。其核心价值在于 CPU-GPU 内存一致性架构，消除传统 PCIe 瓶颈，实现 864GB 统一寻址空间（384GB HBM3e + 480GB LPDDR5X），让 GPU 可以直接访问 CPU 侧大容量内存，无需数据拷贝。

  • 超芯片架构：1 颗 Grace CPU（72 核 Arm Neoverse V2）+ 2 颗 B200 GPU，通过 NVLink-C2C（900 GB/s）实现 CPU-GPU 内存一致性
  • 单模块 GPU 显存：2× 192GB HBM3e = 384GB（单模块），每颗 B200 带宽 8.0 TB/s
  • 统一寻址空间：864GB（384GB HBM3e + 480GB LPDDR5X），CPU 与 GPU 共享统一内存地址空间，消除 PCIe 瓶颈
  • NVLink v5：1.8 TB/s（GPU 间），与 B200 相同；36 颗 GB200 通过 NVLink Switch 构建 72-GPU 全互联域
  • 功耗：约 2700W（GB200 模块整体），必须采用液冷散热
💡 GB200 的核心价值不在于单模块性能，而在于通过 NVLink Switch 构建 72-GPU 全互联域（NVL72）。售前应将 GB200 放在超节点语境下讲解，而非孤立对比单卡。

2.6 GB300（Grace Blackwell Ultra）——2025 推理旗舰超芯片

GB300 是 NVIDIA 2025 年 GTC 大会发布的 Grace Blackwell Ultra 超芯片，是 GB200 的升级版。GPU 从 B200 升级为 B300（Blackwell Ultra），单卡显存从 192GB 提升至 288GB（+50%），FP4 稠密算力提升 1.5×，注意力层（Attention）性能提升 2×。GB300 专为 AI 推理（尤其是测试时计算 / test-time scaling）场景设计，处理 DeepSeek-R1 等推理模型时响应时间从 1.5 分钟缩短至约 10 秒，是 2025—2026 年 AI 推理工厂的旗舰硬件。

  • Blackwell Ultra GPU（B300）：单卡 288GB HBM3e，相比 B200 的 192GB 提升 50%，单卡可装下更大模型，减少多卡分片通信开销

2.8 GPU 服务器 CPU 配置规格（Intel / AMD）

在智算售前方案中，客户常把全部注意力放在 GPU 选型上，而忽视了 CPU 的搭配。实际上，GPU 服务器中 CPU 虽然不直接参与大模型的核心矩阵运算，但承担着数据加载与预处理、推理请求调度、KV Cache 管理、框架运行时调度、以及 PCIe 数据搬运等关键任务。CPU 核心数不足或内存带宽不够，会导致 GPU 在等待数据时出现"数据饥饿"，实际 MFU 大幅下降。因此，针对不同代际的英伟达专业卡，选配正确的 CPU 平台是售前方案的必要环节。

2.8.1 CPU 在 GPU 服务器中的核心职责

  • 数据加载与预处理：训练阶段从存储读取数据样本（图像解码、文本分词、数据增强），CPU 核心数直接决定 DataLoader 的并行能力。经验法则——每个 GPU 至少分配 4~8 个 CPU 物理核用于数据加载，否则 GPU 利用率受限。
  • 推理请求调度与批处理：推理服务（如 vLLM、TGI）的请求队列管理、动态 Batching、Token 分发均由 CPU 完成。高并发场景下 CPU 核心不足会成为吞吐瓶颈。
  • PCIe 数据搬运：PCIe 形态的 GPU（如 L40S、L20、RTX 系列）通过 PCIe 与 CPU 通信，CPU 的 PCIe 通道数和带宽直接影响数据传输效率。SXM 形态的 GPU 通过 NVSwitch 互联，CPU 主要负责控制平面。
  • 内存容量管理：CPU 侧 DDR 内存用于存放操作系统、框架、数据缓存。训练场景下建议 CPU 内存 ≥ GPU 总显存的 1.5~2 倍；推理场景下建议 CPU 内存 ≥ GPU 总显存的 1~1.5 倍。

2.8.7 GB200 的特殊架构：Grace CPU 替代 x86

与 A100/H100/H200/B200 服务器采用 Intel 或 AMD x86 CPU 不同，GB200 超芯片集成了 NVIDIA 自研的 Grace CPU（72 核 ARM Neoverse V2），不再使用 x86 平台。Grace CPU 通过 NVLink-C2C（900 GB/s）与 B200 GPU 实现内存一致性互联，CPU 与 GPU 共享统一地址空间，数据传输无需经过 PCIe，从根本上消除了传统 x86 架构中 CPU-GPU 间的 PCIe 瓶颈。

Grace CPU 的关键规格：72 核 ARM Neoverse V2、480GB LPDDR5X（带宽 512 GB/s）、与 2×B200 共享 864GB 统一寻址空间。这一架构使 GB200 在大模型推理（尤其需要 CPU 参与调度的多并发场景）中相比传统 x86 + GPU 方案有结构性优势，但也意味着软件生态需要适配 ARM 架构。

2.8.8 CPU 配置快速估算口诀

售前现场快速估算 GPU 服务器的 CPU 核心数需求，可使用以下公式：

训练服务器：CPU 总核心数 ≥ GPU 数 × 6~8（每个 GPU 分配 6~8 核用于数据加载与预处理）
推理服务器：CPU 总核心数 ≥ GPU 数 × 4~6（推理请求调度和 Token 分发开销较低）
CPU 内存 ≥ GPU 总显存 × 1.5（训练）或 × 1.0~1.5（推理）

⚠ 注意：以上为经验估算值，实际需求与框架（PyTorch/vLLM/TensorRT-LLM）、数据格式、并发模型强相关。大模型推理（如 70B+ 参数）在多并发下 CPU 消耗会显著上升，建议留 30% 余量。

第四章　服务器形态与超节点架构对比

单卡性能之上，服务器形态是决定客户实际采购单元和部署体验的关键。2025 年，AI 服务器形态已从传统 8 卡服务器演进到超节点（SuperPod），技术路线之争同样激烈。

4.1 标准 8 卡 AI 服务器（HGX/DGX 形态）

这是目前装机量最大的 AI 服务器形态，是大多数训练和推理集群的基本积木单元。以 8×H100 HGX 服务器为例，典型配置如下：

  • GPU：8× NVIDIA H100 80GB SXM（机内通过 4 颗 NVSwitch 实现全互联 full-mesh）
  • CPU：双路高性能 CPU（如 Intel Xeon Sapphire Rapids 或 AMD EPYC Genoa），提供 PCIe Gen5 接口
  • 内存：1TB~2TB DDR5 DRAM（主机内存，CPU 侧）
  • 计算网卡（多轨接入）：8× ConnectX-7 400G IB/RoCE 网卡，每张 GPU 对应一张网卡，上连到 8 台不同的 Leaf 交换机（多轨架构）
  • 存储网卡：1~2 张独立存储网卡（BlueField-3 DPU 或独立 RoCE 网卡），连接高性能并行文件系统
  • 本地存储：8× NVMe SSD，用于临时数据缓存和 Checkpoint 快速存储
  • 整机功耗：约 8–12 kW（GPU 占大头，约 5.6 kW，CPU+内存+网卡+冷却约 3–5 kW）
  • 采购参考价：约 ¥210 万/台（H100 版），H800 约 ¥250 万/台（2023 高峰期，2025 年已下降约 10–20%）

4.2 超节点（SuperPod）——2025 最重要的形态变革

超节点（SuperPod / 超级节点）的出现，是为了解决一个核心矛盾：随着大模型参数规模突破万亿，张量并行（TP）、专家并行（MoE/EP）对机内通信带宽的要求极高，而传统 8 卡服务器内的 NVLink带宽和 PCIe 带宽已成为明显瓶颈。超节点的核心思路是把高速互联总线从机内扩展到机柜级乃至多柜级，构建一个"逻辑上的超大单机"。

超节点的关键优势在于：①更大的共享高带宽域（HBD），张量并行不必跨越延迟更高的网络层；②统一的大容量 HBM 地址空间，MoE 专家路由可以在超节点范围内低延迟访问任意专家参数；③比传统多机多卡方案更低的通信延迟，是训练万亿参数 MoE 大模型的必要基础设施。

4.2.1 NVIDIA GB200 NVL72

NVL72 是英伟达 GB200 NVLink 72-GPU 方案，由 36 颗 GB200 超芯片（每颗含 1 Grace CPU + 2 B200 GPU）组成，通过铜缆直连和 NVSwitch 实现 72 颗 GPU 的全互联。物理形态约 2 个标准 19" 机柜，功耗约 145kW，需要液冷系统。整体系统 BF16 算力约 180 PFLOPS，HBM 容量 13.8TB，是目前单一系统最高算力密度的商用方案。

4.2.2 华为 CloudMatrix 384（CM384）

CM384 由 12 个计算机柜（每柜 4 台 8×910C 服务器）+ 4 个网络机柜（CloudEngine 16800 交换机）构成，共 384 颗昇腾 910C NPU + 192 颗鲲鹏 920 CPU。通过 6912 个 400G 光模块实现全对等光互联，系统 BF16 算力 300 PFLOPS（1.7× NVL72），HBM 容量 49.2TB（3.6× NVL72）。代价是系统总功耗约 600kW（4.1× NVL72），维护复杂度高，且光模块故障率需要"昇腾云脑"实时监控（1分钟感知-3分钟定位-10分钟恢复）。

4.3 大规模 AI 集群（千卡~十万卡）

当训练任务超出单一超节点的规模时，需要把数百甚至数万台服务器/超节点通过高性能网络互联，构成大规模集群。典型代表包括：百度文心大模型集群（约 1.6 万卡 A100）、字节跳动豆包模型集群（已有报道的万卡级 H800 集群）、阿里通义千问集群（HPN 双平面架构，支持超过 24 万 GPU）、腾讯星脉集群（胖树三层，最高 52 万 GPU 理论规模）。

💡 大规模集群的成本已不再由 GPU 价格单独决定，网络（IB 交换机+光模块）、高性能存储、机房建设、电力容量往往各占 10–25% 的总建设成本。

第五章　组网方案全景

组网是智算售前中最能体现技术深度的环节，也是客户技术团队最愿意深入探讨的议题。核心认知：AI 集群算力不随 GPU 数量线性增长，存在"加速比"瓶颈（通常 <1），而降低这个瓶颈的核心技术就是 RDMA + 高性能无损网络。理解了这一点，再去推荐 IB 或 RoCEv2，就有了充分的技术底气。

5.2 Scale-Up 与 Scale-Out：两种扩展维度

5.2.1 Scale-Up（纵向扩展）

Scale-Up 指在单节点或超节点内部，通过高速总线把多个 GPU/NPU 紧密耦合，使其协同表现如一台超大单机。承载张量并行（TP）、序列并行（SP）、专家并行（MoE EP）等对通信延迟极为敏感的计算任务。

  • 时延要求：纳秒级（<200ns），通信开销不可被计算时间掩盖（否则 GPU 空转等待）
  • 带宽要求：TB/s 级超高带宽，英伟达 NVLink v4 单卡 900GB/s、v5 达 1.8TB/s；华为 UB 总线 2.8Tbps/NPU
  • 技术实现：NVLink/NVSwitch（英伟达），UB/HCCS（华为），Infinity Fabric（AMD），UALink（英特尔等联盟）
  • 典型规模：NVL72（72 GPU）、CM384（384 NPU），未来将出现更大规模超节点

5.2.2 Scale-Out（横向扩展）

Scale-Out 指通过网络把多台服务器/超节点连接成大规模集群，承载流水线并行（PP）、数据并行（DP）等可以被计算时间部分掩盖的通信任务。

  • 时延要求：微秒级（~2–5μs），可被计算部分掩盖，但仍需足够低以保证集群效率
  • 带宽要求：单 GPU 接入带宽 100–400Gbps，全集群需 P 级（千 TB/s）交换能力，1:1 无收敛
  • 技术实现：InfiniBand（NDR 400G）、RoCEv2（PFC+ECN+DCQCN）、全调度以太网 GSE（中国移动主导）
  • 典型规模：千卡（RoCEv2 优化）到十万卡以上（IB 或多平面以太网）

5.4 组网拓扑结构详解

5.4.1 CLOS 胖树（Fat-Tree）——通用主流方案

CLOS 胖树是目前应用最广泛的智算集群组网拓扑，由 Leaf 交换机、Spine 交换机、Core 交换机组成（中小规模使用 Spine-Leaf 两层）。特点是无阻塞（参数面 1:1 收敛比）、扩展性好（以 Pod 为单位扩展）、运维成熟度高。接入方式分为：

  • 多轨接入（Rail-Optimized / Multi-Rail）：服务器 8 张网卡分别上连 8 台不同 Leaf 交换机。通信效率高（同机架内部分流量可在 Leaf 层消化，不经 Spine），是大模型训练集群的推荐方案。代价是每台服务器需要 8 张 400G 网卡，布线复杂。
  • 单轨接入（Single-Rail）：服务器全部网卡接入同一台 Leaf 交换机。布线简单，但通信效率低，且单台 Leaf 故障影响全部服务器，适合中小规模或推理场景。

典型案例：腾讯星脉集群（Block 1024卡/Pod 65536卡/Cluster 524288卡理论规模）；阿里 HPN 双平面架构（Segment 1024卡，可扩展至约 24.5 万 GPU）。

5.4.2 多平面胖树（MPFT）——2025 年新主流

多平面胖树（Multi-Plane Fat-Tree，MPFT）由 DeepSeek 在 2025 年 V3 训练论文中提出并验证，核心思路是用多个二层胖树平面取代一个三层胖树，在同等组网规模下实现架构扁平化。DeepSeek 实测验证：在 16384 GPU 的集群中，8 平面 MPFT：

  • 建设成本比三层胖树降低约 40%（减少了大量 Spine 层交换机）
  • 通信时延降低约 30%（减少了一层转发跳数，从 5 跳降到 3 跳）
  • All-to-All 通信性能与单平面多轨方案相当，特别适合 MoE 架构大模型
  • 具备更好的故障隔离能力（某一平面故障不影响其他平面）
💡 多平面方案的挑战：平面间路由切换需要端侧感知（NIC/RDMA 层需要支持跨平面切换），运维复杂度有所提升。但综合性价比已成为 2025 年以后大规模智算集群的主流选择。

5.4.3 Dragonfly+ 拓扑——超大规模最优成本

Dragonfly 及其改进版 Dragonfly+ 是 HPC 领域广泛使用的高性能网络拓扑。每个 Group 内设备通过二层胖树互联，Group 间两两直连（全互联），形成双层结构，显著减少网络跳数和交换机总量。优势：网络直径最小（2~3跳）、时延低、交换机节省约 20%、可支持超过 27 万 GPU（约为三层胖树的 4 倍规模）。劣势：扩展时需重新部署链路，可维护性相对差，HPC 背景较强适合算法工程师理解。
